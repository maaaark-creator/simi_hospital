# 本组契约适配设计说明

本文档面向老师和同学，目的是用尽量简明、易读的方式说明：在课程统一契约已经确定的前提下，本组项目应如何更自然地嵌入全院系统，以及为什么在合并过程中会出现一些结构性困难。

---

## 先看这里

如果只想快速了解本组的结论，可以先看这 7 点：

1. 统一契约主要围绕“患者主流程状态流转”设计，而本组项目更偏“MDT/术前决策能力服务”，两者关注重点不完全相同。
2. 由于本组在最初统一规划阶段没有被完整纳入主流程型模块设计，所以合并时出现的困难，更多来自结构衔接，而不是能力缺失。
3. 本组更适合作为一个标准 `consultation service` 接入系统，而不是直接承担患者主流程状态管理。
4. 本组建议保留现有 `chat` 作为展示与演示入口，同时新增 `api/v1` 作为正式联调入口。
5. 对本组来说，`HTTP` 更适合承担 consultation 调用和人工补充，`Redis` 更适合承担全院事件回流和自动补充触发。
6. 为了更好发挥本组模型能力，需要新增一层“协同补充层”，负责二次补充、有限轮次重评与最终收束。
7. 其他组若希望更充分使用本组能力，最关键的做法是：第一次 consultation 尽量提供结构化摘要，后续关键检查、影像、危急值和病情变化及时回流到本组。

一句话概括：

`本组更适合作为 MDT/术前决策能力服务接入全院系统，由其他组持有患者主流程，本组持续提供综合判断、重评与最终收束能力。`

## 1. 这份说明想解决什么问题

本文档主要回答 5 个问题：

1. 为什么现有统一契约与本组项目在合并时会出现一些衔接困难
2. 在不改变契约主框架的前提下，本组最合适的系统位置是什么
3. `HTTP` 应该从哪里输入，向哪里输出
4. `Redis` 更适合承担什么角色，如何形成信息回流
5. 二次补充、二次问询以及最终收束到手术方案/诊疗手段，应该怎样与其他组协作

## 2. 为什么在合并阶段会遇到一些结构性困难

统一契约 [接口契约_v1.0.md](/d:/hemu_STU/BME/simi_hospital/接口契约_v1.0.md:1) 设计得很完整，已经清楚规定了：

- 患者主流程状态流转
- 跨组转诊
- 入院/出院
- 标准事件信封
- Redis 总线
- LLM 网关
- 基础数据格式

这些内容对于 A/B/C/D 这类“拥有患者主流程状态”的组非常合适。但对本组来说，困难主要来自一个客观原因：

- 本组在最初统一规划阶段没有被完整纳入“主流程型模块”设计
- 而本组后来形成的是一套更偏“复杂病例会诊、二次补充后重评、最终收束到可执行方案”的能力结构

因此，在合并时出现的主要难处，不是本组功能无法接入，而是：

- 契约对本组这种能力型服务的输入模式覆盖较少
- 契约对 consultation 结果对象没有做专门定义
- 契约对“二次补充信息由谁统筹”没有单独建模

从这个角度看，更准确的说法不是“契约不适合本组”，而是：

**现有契约已经提供了全院运行骨架，但对于本组这种 MDT/术前决策型服务，还需要补上一层会诊循环与补充协调逻辑，才能更充分发挥本组模型能力。**

## 3. 本组项目的能力特点

本组项目当前的核心能力，不是主流程状态推进，而是：

- 接收复杂病例信息
- 调用 `MDT_Call -> 各专科 agent -> HeadDoctorAgent`
- 形成跨专科综合判断
- 识别风险和关键缺项
- 在后续补充信息到来后再次评估
- 最终把结果收束到：
  - 手术方案
  - 诊疗手段
  - 或保守处理建议

本组已有代码中，与这部分能力最相关的是：

- [head_doctor/head_doctor_agent.py](/d:/hemu_STU/BME/simi_hospital/head_doctor/head_doctor_agent.py:1)
- [MDT_Call/mdt_call_agent.py](/d:/hemu_STU/BME/simi_hospital/MDT_Call/mdt_call_agent.py:1)
- [final_surgical_plan_memory_agent.py](/d:/hemu_STU/BME/simi_hospital/final_surgical_plan_memory_agent.py:1)
- [mdt_chat_server.py](/d:/hemu_STU/BME/simi_hospital/mdt_chat_server.py:1)

其中，chat 页面和 chat server 目前主要承担：

- 展示
- 演示
- 人工输入病例
- 多轮补充体验

但从系统联调角度看，这些能力更适合被抽象为独立的后端服务能力，而不是继续只挂在 chat 页面上。

## 4. 本组在全院系统中的更合适定位

从统一契约出发，本组更适合被定位为：

- `MDT consultation service`
- `perioperative planning service`
- `re-evaluation service for high-risk cases`

这意味着：

- 其他组持有患者主流程
- 本组持有复杂病例综合判断与收束能力

### 4.1 其他组更适合负责

- 注册 / 分诊 / 路由
- 转诊真正执行
- 入院 / 出院
- 一线数据采集
- 与患者直接交互的主流程

### 4.2 本组更适合负责

- 接收 consultation 请求
- 综合多专科判断
- 输出风险、建议、是否可推进、是否需补充
- 接收补充信息后重新评估
- 在有限轮次后收束为：
  - 完整手术方案
  - 临时诊疗手段
  - 保守延迟处理建议

这种分工更有利于发挥本组现有模型结构的优势，而不必让本组去重复实现其他组已经承担的主流程功能。

---

## 5. 本组建议采用的整体嵌入逻辑

对本组这套仓库，更适合的接法是：

- 保留 `chat`：负责展示、演示、人工输入病例
- 新增 `api/v1`：负责全院联调与标准 consultation 能力
- 新增 `redis adapter`：负责接收全院事件、触发自动补充与必要的再评估
- 保留 `HeadDoctorAgent`：继续作为核心决策引擎
- 新增“协同补充层”：负责二次补充、轮次控制、最终收束

这样的好处是：

- 不需要推翻现有项目主体结构
- 可以继续保留本组在多专科综合判断和术前方案收束上的特点
- 可以较平稳地接入统一契约
- 能让本组能力服务于其他组，而不是只停留在前端演示页面

### 6.2 三条信息线

这个架构里有 3 条线：

- **主流程线**：患者在 A/B/C/D 组之间流动
- **能力调用线**：其他组通过 HTTP 向本组请求 MDT 评估
- **信息回流线**：后续新信息通过 Redis 事件或 HTTP 补充回到本组，再次重评

本组主要工作在线 2 和线 3 上，不直接拥有线 1。

## 7. HTTP 应该怎么设计

### 7.1 HTTP 的角色

在本组这里，HTTP 更适合承担：

- consultation 调用
- 补充信息上传
- consultation 结果查询

也就是说，HTTP 在本组不是主状态流转接口，而是“能力调用接口”。

### 7.2 推荐接口清单

本组建议提供 4 个接口：

```text
POST /api/v1/consultations/mdt
GET  /api/v1/consultations/{consultation_id}
POST /api/v1/consultations/{consultation_id}/supplements
GET  /api/v1/health
```

### 7.3 每个接口的输入与输出

#### `POST /api/v1/consultations/mdt`

用途：

- 由当前持有患者主流程的组，把复杂病例交给本组做 MDT / 术前评估

输入来源：

- A/B/C/D 组
- 或演示层先把自然语言整理成结构化后再调用

输出去向：

- 直接返回给调用方

调用方收到后，可以据此决定：

- 是否继续当前路径
- 是否补充必要信息
- 是否转科 / 转诊
- 是否进入 ICU / 病房 / 术前准备等下一阶段

#### `GET /api/v1/consultations/{consultation_id}`

用途：

- 当 consultation 采用异步执行时，供调用方查询结果

#### `POST /api/v1/consultations/{consultation_id}/supplements`

用途：

- 上传二次补充信息
- 触发重新评估或最终方案收束

输入来源：

- 主责组补录
- 其他科室整理后的补录
- 本组内部 patient agent / doctor agent 整理后的结构化补录

输出去向：

- 返回新的 consultation 结果

#### `GET /api/v1/health`

用途：

- 联调和部署健康检查

## 8. Redis 更适合承担什么角色

### 8.1 Redis 的角色

Redis 在本组这里不适合承担 consultation 全量返回，更适合承担：

- 接收全院状态变化
- 自动补充病例快照
- 触发再次评估
- 发布必要的告警和摘要

这样设计的原因是：

- consultation 结果通常结构较复杂，更适合通过 HTTP 点对点返回
- Redis 更适合表达“新信息来了”“病情变了”“现在值得重评了”这一类触发

### 8.2 建议订阅的事件

优先订阅：

- `hospital.patient.triaged`
- `hospital.patient.transferred`
- `hospital.patient.admitted`
- `hospital.vital.critical`
- `hospital.lab.resulted`
- `hospital.imaging.resulted`

这些事件分别可以帮助本组做：

- `patient.triaged`
  - 初次识别高风险病例，必要时自动创建 consultation

- `patient.transferred`
  - 更新病例当前场景，决定是否需要轻量重评

- `patient.admitted`
  - 患者进入 ICU/病房后，触发围术期或病房阶段的再评估

- `vital.critical`
  - 高优先级触发重评

- `lab.resulted`
  - 自动填补化验缺口

- `imaging.resulted`
  - 自动填补影像缺口

### 8.3 建议发布的事件

必须支持：

- `hospital.alert.raised`

可谨慎扩展：

- `hospital.agent.mdt_completed`
- `hospital.agent.mdt_recommended`

建议策略：

- 标准联调优先依赖契约已经定义好的事件
- 本组扩展事件只做摘要广播，不作为主链依赖

## 9. 本组最重要的新角色：协同补充层

这部分是本组真正需要补上的“中间层”。

统一契约中已经有主流程事件、有 HTTP 接口、有状态机，但没有单独定义：

- 谁来接住后续补充信息
- 谁来合并这些信息
- 谁来判断哪些信息已经足够触发重评
- 谁来判断何时不再继续追问，而应该开始收束

因此，本组建议新增一个后端角色：

```text
Supplement Coordinator / FinalPlanCoordinator
```

它更像：

- 医院信息部
- 病案整合中台
- 会诊秘书台

而不是 chat 页面。

### 9.1 它负责什么

它负责 6 件事：

1. 保存 consultation 对应的完整病例快照
2. 收集后续补充信息
3. 维护“还缺什么”
4. 区分必要补充和可选补充
5. 判断是否值得再跑一轮评估
6. 在达到阈值后收束为完整方案或保守处理建议

### 9.2 它和现有代码的关系

当前本组已有两段接近这种能力的逻辑：

- [mdt_chat_server.py](/d:/hemu_STU/BME/simi_hospital/mdt_chat_server.py:455) 中的 `session + supplements + last_workflow`
- [final_surgical_plan_memory_agent.py](/d:/hemu_STU/BME/simi_hospital/final_surgical_plan_memory_agent.py:73) 中的 `update_and_plan(...)`

现在更适合做的不是推倒重来，而是把这两段从 chat 编排里抽象出来，正式变成：

- consultation snapshot 管理
- supplement 协调
- final plan 收束

## 10. 本组 consultation 输出给谁，他们如何配合

这是本组能否真正发挥价值的关键。

### 10.1 输出给谁

本组的 HTTP consultation 结果，主要输出给：

- 当前持有患者主流程的组

例如：

- 急诊病例，主要输出给急诊组
- ICU 中高风险病例，主要输出给 ICU 组
- 住院阶段术前准备病例，主要输出给住院组

### 10.2 他们如何配合，才能发挥本组能力

调用方最理想的协作方式是：

1. **第一次 consultation 时尽量提供结构化摘要**
   - 患者基本信息
   - 当前场景
   - 当前主诉与初步诊断
   - 生命体征
   - 已知关键化验/影像/检查
   - 当前希望本组回答的问题

2. **后续新信息及时回流**
   - 危急值
   - 关键影像结果
   - 关键化验结果
   - 转入 ICU/病房后的状态变化
   - 围术期准备条件是否满足

3. **把本组输出当作综合判断依据**
   - 用来支持是否继续当前路径
   - 是否补充信息
   - 是否转科/转诊
   - 是否进入术前准备或监护阶段

也就是说，本组最适合做的是：

- 给出综合判断
- 告诉主责组“哪些最值得补”
- 在补充信息回流后重新评估

而不是直接替其他组执行主流程动作。

## 11. 二次补充与二次问询，应该怎样与其他组协作

### 11.1 总原则

在统一架构下，更建议这样分工：

- 本组不直接接管其他组的患者主流程
- 本组不直接改别组状态
- 本组主要负责：
  - 告诉主责组“哪些信息最值得补”
  - 在主责组补完或新事件到来后重新评估

### 11.2 二次补充的两种来源

二次补充来源分成两类：

#### A. 自动补充

来自 Redis 事件：

- 新化验
- 新影像
- 新生命体征
- 转科 / 入院

这是第一优先级。

优点是：

- 不增加人为沟通负担
- 最符合契约现有架构
- 最有利于联调
- 最能把本组模型从“一次性判断”提升为“持续感知关键新证据后再判断”

#### B. 人工补充

来自主责组通过 HTTP 再次调用：

- 医生补录
- 护士补录
- 病历系统补录
- 本组内部 patient agent / doctor agent 整理后的结构化补录

这类补充统一走：

```text
POST /api/v1/consultations/{consultation_id}/supplements
```

### 11.3 为什么不建议做成跨组多轮对话

为了降低系统复杂度，不建议做成：

- 本组直接去 HTTP 调别组追问
- 本组和每个组之间形成多轮循环对话

因为这样会带来几个问题：

- 责任边界不够清楚
- 信息循环容易失控
- 调试和联调成本明显增加

更合适的责任边界是：

- 本组负责列出 `required_updates`
- 主责组负责把这些信息补回来

如果后续课程允许增加更智能的协作层，也可以在主责组内部用 patient agent / doctor agent 去帮助完成这一步，但跨组边界上仍建议只传结构化补充结果。

## 12. 如何把二次补充最终收束为手术方案或诊疗手段

这是本组架构里最重要、也最需要单独说清的一点。

### 12.1 收束不意味着必须等所有信息都齐

如果系统一直停留在“继续补信息”，本组模型在复杂病例上的价值就很难体现出来。  
因此，本组建议允许 3 种收束结果：

#### 结果 A：完整手术方案

适用情况：

- 关键阻塞项已基本解决
- 可以按术前 / 术中 / 术后给出步骤

输出内容：

- `perioperative_plan`
- `readiness = ready` 或 `needs_optimization`

#### 结果 B：临时诊疗方案 / 保守处理方案

适用情况：

- 信息仍不完全
- 但不适合继续无限等待

输出内容：

- `provisional_plan`
- `required_updates`
- `unknown_risks`
- `next_action = stabilize_then_reassess`

#### 结果 C：明确不应推进

适用情况：

- 当前风险过高
- 即使继续问，也不适合立即进入手术 / 进一步侵入性路径

输出内容：

- `readiness = contraindicated`
- `recommendations = 先稳定，再转上级监护或继续观察`

### 12.2 谁来决定收束

更适合由后端的 `FinalPlanCoordinator` 来决定，而不是由 chat 页面决定。

推荐依据包括：

- 当前轮次 `round_count`
- `required_updates`
- `unknown_risks`
- `current_status_level`
- 新补充信息是否已经解决核心阻塞

### 12.3 现有能力如何承接

你们现有代码已经有两个很重要的基础：

- [head_doctor/head_doctor_agent.py](/d:/hemu_STU/BME/simi_hospital/head_doctor/head_doctor_agent.py:1)
  - 负责完整综合评估

- [final_surgical_plan_memory_agent.py](/d:/hemu_STU/BME/simi_hospital/final_surgical_plan_memory_agent.py:73)
  - 负责“已有 MDT 结果 + 补充信息 -> 最终方案”

因此，推荐承接方式是：

- 第 1 轮：走 `HeadDoctorAgent.evaluate_case(...)`
- 第 2 轮及以后：
  - 若信息变化大，走 `full_re_evaluation`
  - 若只是解决阻塞，走 `plan_update`

### 12.4 推荐的收束阈值

建议采用这样的有限轮次规则：

- `round_count = 0`
  - 首次完整 MDT

- `round_count = 1`
  - 接收第一轮补充后重评

- `round_count >= 2`
  - 不再无限追问
  - 若仍信息不足，则输出：
    - `provisional_plan`
    - `unknown_risks`
    - `required_updates`
    - `best_safe_next_step`

这样既保留了本组模型的多轮重评优势，也保证了流程可以闭环。

## 13. HTTP 与 Redis 的职责分工

### HTTP

更适合负责：

- 首次 consultation 输入
- 人工补充输入
- consultation 结果查询

特点：

- 点对点
- 结果可全量返回
- 适合复杂结构

### Redis

更适合负责：

- 全院状态变化广播
- 自动补充信息输入
- 告警
- 必要摘要广播

特点：

- 广播式
- 更适合标准化摘要与触发信号
- 不适合承担 consultation 全量结果主链

一个最重要的原则是：

**HTTP 是能力调用通道，Redis 是状态回流通道。**

这样分工，也最有利于其他组稳定配合本组发挥能力。
