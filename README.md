# 基于 MDT 调度机制的多智能体术前评估与会诊决策原型系统。



本项目面向复杂外科病例的术前评估、专科协同和综合决策场景，构建了一个以 `MDT_Call` 为分诊入口、以各专科 Agent 为执行单元、以 `head_doctor` 为最终整合节点的多智能体系统。它并不是把多个模型简单并排调用，而是尝试按照真实医院会诊链路组织信息流和责任流：先分诊，再评估，再追问，最后汇总形成建议。

系统当前支持麻醉科、心内科、肝胆胰外科、神经外科、骨科等专科模块，并提供完整命令行演示脚本、根目录统一配置、以及可视化聊天页面，适合用于课程展示、流程原型验证和多学科决策研究。

```text
├─ multi_agent_config.py                   # 根目录默认配置与配置加载器
├─ multi_agent_config.json                 # 运行模式配置
├─ MDT_Call/                              # MDT 分诊与二次追问模块
├─ head_doctor/                           # 总控整合模块，形成最终建议
├─ Anesthesia/                            # 麻醉科 Agent
├─ Cardiology/                            # 心内科 Agent
├─ Hepatobiliary and Pancreatic Surgery/  # 肝胆胰外科 Agent
├─ Neurosurgery/                          # 神经外科 Agent
└─ Orthopaedics/                          # 骨科 Agent
```

各专科目录基本采用一致的分层方式：

- `patient_mapper_agent.py` 或对应 mapper：负责病例结构化与字段标准化。
- `kb_retriever.py` 或对应 retriever：负责专科知识检索。
- `*_decision_agent.py`：负责形成专科结论。
- `*_kb.json`：专科知识库。
- `run_*_pipeline_demo.py`：单模块演示脚本。

这种组织方式保证了每个专科模块都具备相对独立的知识、推理和输出接口，便于扩展新专科，也便于在总控流程中统一接入。

## 2. 流程与模块展示

项目当前主流程如下：

```text
患者输入
  -> MDT_Call 首轮分诊
  -> 触发相关专科 Agent
  -> 形成初始专科意见
  -> head_doctor 汇总不确定点
  -> MDT_Call 二次追问与补充派发
  -> 专科返回澄清意见
  -> head_doctor 综合输出最终建议
```

对应到代码层面：

- `mdt_chat_server.py`：面向交互式输入，适合展示完整工作流。
- `MDT_Call/mdt_call_agent.py`：负责首轮分诊和后续追问任务派发。
- `head_doctor/head_doctor_agent.py`：负责调度专科、识别存疑点、整合多轮意见。
- 各专科 Agent：负责专科视角下的结构化、检索和决策。

在当前实现中，`head_doctor` 会先调用 `mdt_call` 做首轮分诊，再根据返回的专科列表逐个触发相应 Agent；若首轮结论中存在缺失信息、风险提示或检索空缺，再回流 `mdt_call` 进行二次追问；最后由 `head_doctor` 生成结构化综合意见。这个闭环是本项目区别于普通问答式医疗 Demo 的关键。

## 4. 项目背景与现实意义

复杂外科病例的诊疗决策，本质上不是单一科室可以独立完成的任务。患者能否手术、应当采用何种手术方式、是否需要先进行围术期优化、有哪些高风险因素、哪些信息仍然缺失，往往同时受到原发疾病、合并症、麻醉风险、器官功能状态和患者整体耐受性的共同影响。现实医院中的重大病例讨论，本来就依赖多学科协作。

但传统会诊流程普遍存在几个问题：第一，会诊触发依赖人工经验，何时发起、邀请哪些科室参与，常常缺乏稳定标准；第二，病例信息分散在病历、检验、影像和口头讨论中，不同专科接收到的信息并不总是完整一致；第三，会诊结论往往停留在一次性讨论，缺乏对不确定点的结构化追问和再整合；第四，不同专家意见之间的冲突、风险优先级和执行顺序，通常需要资深医生再次人工收束。

本项目的意义就在于把上述问题转化为一个可验证、可复用的流程原型。项目不试图替代临床医生，而是把会诊中的关键环节拆解为清晰的系统职责：谁负责首轮分诊，谁负责专科判断，谁负责补充追问，谁负责最终综合。这样做的价值不在于“让模型说得更多”，而在于让复杂病例决策过程具备更强的结构化表达、可追踪性和可扩展性。

从教学和研究角度看，这种原型同样具有现实意义。它能够把抽象的 MDT 协作机制具体化为一个可运行的系统，使课程展示不再停留在“多个模型同时给答案”的层面，而是能够清楚说明临床决策是如何形成、如何修正、如何收束的。  

相关的背景和意义说明已经单独整理到文件 [MDT驱动的智能术前会诊系统必要性与应用价值分析](MDT驱动的智能术前会诊系统必要性与应用价值分析.md)

## 5. 已实现功能

### 5.1 MDT 首轮分诊

`MDT_Call` 模块负责根据病例首轮信息决定优先触发哪些专科 Agent。它既支持本地检索与规则构建分诊计划，也支持通过模型生成分诊任务。这样做的作用，是把“是否需要某专科介入”本身纳入系统决策，而不是默认所有专科同时参与。

### 5.2 专科独立评估

每个专科 Agent 都遵循“结构化输入 -> 知识检索 -> 形成结论”的统一链路。当前已接入模块包括：

- 麻醉科：围术期风险评估、麻醉相关条件判断与方案支持。
- 心内科：围术期心血管风险识别与相关诊疗建议。
- 肝胆胰外科：胆道感染、胆道减压等相关外科场景判断。
- 神经外科：神经外科术前评估与术式讨论。
- 骨科：骨折、创伤、围手术期风险与手术方案评估。

这种设计确保专科意见来源明确，便于解释“某条结论为什么由这个专科给出”。

### 5.3 不确定点识别与二次追问

`head_doctor` 在收集首轮专科意见后，会主动整理以下类型的不确定项：

- 缺失的关键临床信息。
- 专科识别出的高风险点。
- 检索阶段暴露的信息空缺。

这些内容不会被直接忽略，而是会回流给 `MDT_Call`，由其按问题类型再次分派给最合适的专科进行补充说明。这意味着系统具备多轮会诊能力，而不是一次生成即结束。

### 5.4 最终综合决策

`head_doctor` 负责汇总专科意见、存疑点和追问结果，生成最终结构化输出。当前输出中已经包含：

- `case_summary`
- `final_plan`
- `key_risks`
- `next_steps`
- `unresolved_issues`
- `medical_record_plan`
- `safety_boundary`

在 API 模式下，`head_doctor` 还支持多模型审阅后再统一合成最终意见，使最终结论不只是简单拼接，而是带有明确的冲突处理和风险优先级判断。

#### 分层架构的直接优势

- 展示层、调度层、专科层、知识层彼此分工明确，便于答辩时解释系统职责边界。
- 各专科采用统一内部接口，新增专科时只需补齐 `mapper + retriever + decision + kb` 四类组件。
- 总控逻辑和专科逻辑分离，避免把所有规则写进一个中心脚本，降低维护复杂度。
- 前端进度展示可以直接映射到后端分层流程，使系统具备较强的可解释性和可观测性。

## 6. 项目优势

### 6.1 技术上的优势

第一，项目采用模块化分层架构，而不是把结构化、检索、决策和总控逻辑全部堆叠在一个脚本中。每个专科都有相对稳定的接口，便于扩展、替换和维护。

第二，系统支持本地启发式模式与 API 模式双轨运行。这样既能在无外部依赖时验证流程，也能在需要更强语义能力时引入模型服务，兼顾了可演示性和可扩展性。

第三，知识检索与专科决策被明确拆分。相比“直接把病例发给大模型生成结论”的方式，这种设计更容易解释系统结论的来源，也更利于后续迭代知识库。

第四，`head_doctor` 与 `MDT_Call` 构成了两级控制结构：前者负责综合判断，后者负责任务分诊。这种职责划分清晰，避免总控模块既当裁判又当执行者造成逻辑混乱。

第五，系统具备流程级可观测性。通过命令行日志和前端实时进度展示，开发者可以看到每一步发生了什么，这对于调试多智能体系统尤为重要。

### 6.2 思路上的优势

第一，本项目强调“决策流程的真实性”，而不是“回答文本的丰富性”。它关注的是会诊如何被触发、专科如何被调用、疑点如何被追问、结论如何被整合，这比单纯比较模型输出长短更接近真实医疗协作。

第二，项目把不确定性当作系统要处理的对象，而不是要掩盖的缺陷。首轮结论中的信息不足和风险冲突不会被直接抹平，而是通过二次分诊机制显式暴露并继续处理。这一点非常关键，因为真实临床决策本来就是在不完全信息下逐步收敛的过程。

第三，项目在结构上保留了临床责任链条。专科负责专科判断，`MDT_Call` 负责分诊调度，`head_doctor` 负责最终整合。这样的设计比“单个大模型包办一切”更符合医院场景下的组织逻辑，也更便于后续扩展出患者因素、状态监测和持续再评估等能力。

第四，这一方案具有天然的扩展性。只要遵循现有的 mapper、retriever、decision 三层接口，就可以继续接入新的专科、新的知识库和新的总控策略，而不必推翻整个系统。

## 7. 当前定位与边界

本项目当前更适合作为以下场景的原型系统：

- 课程项目与答辩展示
- MDT 会诊流程建模
- 多智能体协作机制验证
- 专科知识库与调度策略研究

同时也需要明确其边界：

- 项目当前是教学与研究导向的原型，不构成临床决策依据。
- 实际医疗场景还涉及影像、病理、护理、ICU、药学、康复等更多角色，当前版本尚未全面纳入。
- API 模式依赖外部模型服务，运行效果受网络、服务状态和返回质量影响。

正因为明确这些边界，本项目的价值才更加清晰：它不是“替代医生”的系统，而是一个将复杂会诊流程结构化、可视化、可实验化的工程原型。
